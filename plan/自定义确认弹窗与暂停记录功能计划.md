# 自定义确认弹窗 + 暂停记录功能计划

## 背景

当前应用还有几处使用浏览器原生 `confirm()` 的确认弹窗，例如学习历史页面删除记录、删除标签、删除寄语。这类原生弹窗与 Electron 应用现有 UI 风格不一致，也无法跟随主题样式。

另外，当前学习记录只有“开始 / 结束”两种状态。如果用户学习中途临时休息、接电话或处理其他事情，无法暂停计时，最终统计会把非学习时间也计入学习时长。

## 目标

1. 将删除确认等原生弹窗统一替换为应用内自定义提示弹窗。
2. 评估并实现学习记录暂停 / 继续功能，让有效学习时长更准确。
3. 页面与交互保持简洁、美观、易上手，不增加用户理解成本。
4. 所有涉及前端页面和交互设计的改动，均遵循 `web-design-engineer` 技能的设计方法与质量标准。

## 前端设计原则

本次改动虽然功能范围不大，但涉及用户直接感知的弹窗、按钮和记录状态，因此需要按前端设计任务处理。

### 总体原则

- **简洁**：只呈现当前操作必须理解的信息，不堆叠说明、不制造视觉噪音。
- **美观**：新增元素应融入当前应用视觉体系，和已有 Modal、按钮、卡片、主题风格保持一致。
- **易上手**：按钮文案直接表达操作含义，例如“暂停”“继续”“删除”“取消”，避免只用图标表达关键动作。
- **低干扰**：确认弹窗只打断高风险操作；暂停状态提示要清晰，但不能压过计时主信息。
- **一致性**：颜色、圆角、阴影、间距、动效优先复用现有样式变量和组件习惯，不单独发明一套视觉语言。

### `web-design-engineer` 执行要求

后续真正实施前端改动时，需要遵循以下流程：

1. 先观察现有界面的视觉词汇：颜色比例、按钮状态、Modal 层级、卡片密度、动效方式。
2. 在编码前声明本次 UI 的设计决策，包括：色彩、字号层级、间距、圆角、阴影、动效和状态反馈。
3. 对确认弹窗、暂停按钮、暂停状态提示先做轻量 v0 设计判断，确保方向是“克制、清晰、融入现有产品”。
4. 避免 AI 风格套路：不使用无意义渐变、emoji 堆砌、花哨图标、过度动画或额外装饰。
5. 所有交互组件至少覆盖默认、悬停、点击、聚焦、禁用或防重复点击等必要状态。
6. 完成后需要进行设计自检：视觉层级是否清晰、间距是否统一、按钮含义是否明确、是否存在多余元素、是否在主题切换下仍协调。

## 当前代码现状

### 已有自定义弹窗基础

项目已有完整的 Modal 系统，可以复用：

- `showModal(id)` / `hideModal(id)` 管理弹窗显示隐藏。
- `message-modal` 已提供自定义消息提示。
- 样式已有 `.modal-root`、`.modal-backdrop`、`.modal-panel`、`.modal-footer` 等通用结构。
- 弹窗样式可跟随现有主题。

### 当前原生确认弹窗位置

主要位于 [renderer.js](../renderer.js)：

1. 学习历史删除记录：`historyDelete(id)` 中使用 `confirm()`。
2. 删除标签：标签管理逻辑中使用 `confirm('确认删除该标签吗？相关记录将解除关联。')`。
3. 删除寄语：寄语管理逻辑中使用 `confirm('确认删除该寄语吗？')`。

### 当前学习记录流程

- [main.js](../main.js) 中通过 `ipcMain.handle('study:start-session')` 创建学习记录。
- [main.js](../main.js) 中通过 `ipcMain.handle('study:end-session')` 结束学习记录并计算 `duration`。
- [preload.js](../preload.js) 通过 `contextBridge` 暴露 `startSession()` / `endSession()`。
- [renderer.js](../renderer.js) 中的 `startSession()` / `endSession()` 控制 UI 状态与计时器。
- 数据表 `study_sessions` 当前主要字段为：`start_time`、`end_time`、`duration`、`note`、`created_at`。

当前没有暂停 / 恢复相关状态。

## 实施方案一：自定义确认弹窗

### 推荐做法

新增一个通用 `showConfirm(options)` 方法，返回 `Promise<boolean>`，用于替换原生 `confirm()`。

设计上采用轻量确认卡片，不做复杂信息堆叠：

- 标题一句话说明风险，例如“删除学习记录”。
- 正文只说明后果，例如“删除后无法恢复”。
- 主按钮使用明确动词，例如“删除”。
- 次按钮使用“取消”，位置和样式与现有 Modal 保持一致。
- 危险操作使用克制的危险色，不使用强烈闪烁、夸张图标或 emoji。

调用形式示例：

```js
const confirmed = await showConfirm({
  title: '删除确认',
  text: '确认删除该学习记录吗？',
  okText: '删除',
  cancelText: '取消',
  type: 'danger'
})

if (!confirmed) return
```

### 修改点

#### 1. 修改 [index.html](../index.html)

新增 `confirm-modal`：

```html
<div id="confirm-modal" class="hidden modal-root">
  <div id="confirm-modal-backdrop" class="modal-backdrop"></div>
  <div id="confirm-modal-content" class="modal-panel modal-center modal-sm">
    <div class="modal-body text-center">
      <div id="confirm-title" class="text-lg font-bold mb-2">确认</div>
      <div id="confirm-text" class="text-sm opacity-80 mb-4"></div>
    </div>
    <div class="modal-footer modal-footer-2col">
      <button id="confirm-cancel-btn" class="btn btn-secondary">取消</button>
      <button id="confirm-ok-btn" class="btn btn-danger">确认</button>
    </div>
  </div>
</div>
```

#### 2. 修改 [renderer.js](../renderer.js)

新增：

- `showConfirm(options)`。
- `resolveConfirm(result)` 或等价内部函数。
- 确认按钮、取消按钮、背景点击事件绑定。

需要注意：

- 每次关闭弹窗后清理当前 resolver，避免重复 resolve。
- 如果已有确认弹窗打开，再次调用时应先安全关闭或覆盖当前回调。
- 按钮样式根据 `type` 切换，例如 `danger` 使用危险色。

#### 3. 替换所有 `confirm()`

替换范围：

- 学习历史删除记录。
- 标签删除。
- 寄语删除。

预期效果：所有删除确认都使用统一自定义弹窗。

#### 4. 修改 [styles.css](../styles.css)

优先复用现有样式，只在必要时补充：

- 两列 footer 布局。
- 危险按钮样式（如果现有没有 `btn-danger`）。
- 确认弹窗文本间距。

## 实施方案二：暂停记录功能

### 可行性结论

暂停记录功能可行，且不需要重构现有主流程。

推荐先实现“简单可靠版”：在 `study_sessions` 表中记录暂停状态和累计暂停时长。

### 数据设计

在 `study_sessions` 增加两个字段：

```sql
paused_at TEXT
paused_duration INTEGER DEFAULT 0
```

含义：

- `paused_at`：当前是否处于暂停状态。为 `NULL` 表示未暂停；有值表示暂停开始时间。
- `paused_duration`：累计暂停秒数。

最终有效学习时长：

```text
end_time - start_time - paused_duration
```

如果结束时仍处于暂停状态，需要把最后一段暂停时间也计入 `paused_duration` 后再计算最终 `duration`。

### 修改点

#### 1. 修改 [main.js](../main.js)

新增数据库迁移逻辑：

- 给 `study_sessions` 添加 `paused_at`。
- 给 `study_sessions` 添加 `paused_duration`。

新增 IPC：

```js
ipcMain.handle('study:pause-session', async (_, { sessionId, pausedAt }) => {})
ipcMain.handle('study:resume-session', async (_, { sessionId, resumedAt }) => {})
```

暂停逻辑：

- 校验 session 存在。
- 校验未结束。
- 如果已经暂停，直接返回当前状态。
- 写入 `paused_at`。

恢复逻辑：

- 查询 `paused_at` 和 `paused_duration`。
- 计算本次暂停秒数。
- 累加到 `paused_duration`。
- 清空 `paused_at`。

结束逻辑需要调整：

- 读取 `paused_at` 和 `paused_duration`。
- 如果结束时仍暂停，先补上从 `paused_at` 到 `end_time` 的暂停秒数。
- `duration` 计算扣除累计暂停时长。
- 防止 duration 小于 0。

#### 2. 修改 [preload.js](../preload.js)

在 `window.studyRecord` 中新增：

```js
pauseSession: (payload) => ipcRenderer.invoke('study:pause-session', payload),
resumeSession: (payload) => ipcRenderer.invoke('study:resume-session', payload),
```

如果增加快捷键，也需要处理对应的全局快捷键事件。

#### 3. 修改 [renderer.js](../renderer.js)

新增状态：

```js
let isPaused = false
```

新增函数：

- `pauseSession()`
- `resumeSession()`
- `togglePauseSession()`
- `updatePauseButtonState()`

UI 行为：

- 空闲状态只显示“开始记录”，降低主界面按钮数量。
- 开始记录后隐藏“开始记录”，显示“暂停 / 继续”和“结束记录”。
- 未暂停时按钮显示“暂停”。
- 暂停后按钮显示“继续”。
- 暂停时计时展示冻结或显示“已暂停”。
- 结束记录后恢复只显示“开始记录”，并重置暂停状态。

前端设计要求：

- 暂停按钮应靠近开始 / 结束操作，但不能造成“开始、暂停、结束”三者主次混乱。
- 暂停状态需要一眼可识别，例如按钮文案变为“继续”、计时区域增加低干扰状态文本。
- 不使用只靠颜色表达状态的设计，文案也要明确说明当前状态。
- 动效以轻微过渡为主，例如透明度、背景色或阴影变化，避免分散注意力。
- 小窗口和迷你窗口下仍要保持触控目标清晰、按钮不拥挤。

计时逻辑：

- 当前前端今日时长显示需要扣除暂停时间。
- 可以通过后端返回的 `paused_duration` 与 `paused_at` 辅助计算。
- 为降低复杂度，暂停时可以冻结前端显示，恢复后重新加载历史和今日统计。

#### 4. 修改 [index.html](../index.html)

在开始 / 结束按钮附近增加暂停按钮：

```html
<button id="pause-btn" class="btn btn-warning hidden">暂停</button>
```

#### 5. 修改 [styles.css](../styles.css)

新增或复用按钮状态样式：

- `btn-warning` 或暂停态样式。
- 暂停状态下计时数字弱化样式，例如灰色、低透明度或“已暂停”标识。

### 全局快捷键

采用 `Ctrl+Alt+P` 作为暂停 / 继续切换快捷键：

- [config.js](../config.js) 中新增 `shortcuts.pause = 'Ctrl+Alt+P'`。
- [main.js](../main.js) 注册后发送 `global-shortcut: 'togglePause'`。
- [renderer.js](../renderer.js) 仅在有进行中记录且暂停按钮可用时执行 `togglePauseSession()`。

该快捷键不与现有快捷键冲突：`Ctrl+Alt+S` 开始、`Ctrl+Alt+E` 结束、`Ctrl+Alt+Q` 安全退出、`Ctrl+Alt+M` 迷你模式、`Ctrl+Alt+T` 主题切换。

## 边界情况

需要重点处理：

1. 暂停后直接结束记录。
2. 暂停后关闭窗口到托盘。
3. 暂停后安全退出。
4. 应用重启后恢复未结束 session 的暂停状态。
5. 连续点击暂停 / 继续按钮。
6. 数据库旧数据没有新增字段。

## 验证清单

### 自定义弹窗

1. 打开学习历史，点击删除记录，出现自定义确认弹窗。
2. 点击取消，不删除记录。
3. 点击确认，删除记录并显示 toast。
4. 删除标签、删除寄语也使用同样风格弹窗。
5. 切换主题后弹窗样式正常。
6. 弹窗文案简短明确，用户能在 1 秒内判断“取消”与“删除”的区别。
7. 弹窗视觉层级清晰：标题、说明、主按钮、次按钮的权重合理，不显拥挤。

### 暂停功能

1. 开始记录后，暂停按钮显示。
2. 点击暂停后，按钮变为“继续”，计时停止增长或显示暂停状态。
3. 等待一段时间后点击继续，计时恢复。
4. 结束记录后，最终时长扣除暂停时间。
5. 暂停状态下直接结束记录，最终时长仍正确。
6. 重启应用后，未结束记录和暂停状态能正确恢复。
7. 暂停 / 继续入口易发现，用户不需要阅读说明即可理解如何操作。
8. `Ctrl+Alt+P` 能在记录中切换暂停 / 继续，空闲时不触发误操作。
9. 空闲状态主界面只显示“开始记录”，记录中只显示“暂停 / 继续”和“结束记录”。
10. 暂停状态在日间、夜间、护眼主题下都清晰可辨。
11. 小窗口或迷你模式下，按钮布局不拥挤、不误触。

### 前端设计自检

1. 是否遵循现有视觉体系，而不是新增一套突兀样式。
2. 是否只保留必要信息，没有多余装饰或冗余说明。
3. 所有按钮文案是否清晰表达动作与后果。
4. hover / active / focus / disabled 状态是否完整。
5. 动效是否克制、自然，不影响学习记录主流程。
6. 是否通过浏览器或 Electron 实际界面检查，没有控制台错误、布局溢出或主题异常。

## 建议执行顺序

1. 实施前先按 `web-design-engineer` 要求声明本次 UI 设计决策：色彩、层级、间距、状态反馈和动效。
2. 先实现自定义确认弹窗并替换 3 处 `confirm()`。
3. 手动验证删除确认流程与弹窗视觉质量。
4. 再实现数据库字段迁移和暂停 / 恢复 IPC。
5. 接入 preload 与 renderer UI。
6. 手动验证暂停 / 继续 / 结束 / 重启恢复场景。
7. 做一次前端设计自检，确认页面简洁、美观、易上手。
8. 如一切稳定，确认 `Ctrl+Alt+P` 暂停 / 继续快捷键与现有快捷键不冲突。
